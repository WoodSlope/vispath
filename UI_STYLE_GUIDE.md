# VisPath UI Style Guide

## 规范职责

本文件负责产品视觉语言、信息层级、文案和体验原则。可执行的颜色 Token、间距、组件尺寸、状态矩阵、弹窗契约和验收命令统一维护在 [`DESIGN_SYSTEM.md`](./DESIGN_SYSTEM.md)。进行页面级视觉调整时，必须同时读取两份文档，不能只根据本文件或当前截图判断统一性。

## 适用范围

VisPath 是面向视觉创作者的本地 AI 视觉方向工作台。页面核心路径是：输入原始提示词或参考图 -> 建立探索 -> 比较提示词方案 -> 提交生成 -> 查看结果历史。

这是工具型工作台，不是营销落地页。界面优先保证扫描、比较、输入和恢复操作的效率。

## 视觉语言

- 明亮、克制、纸张感的中性底色。
- 墨色用于主要文字和结构边界，橙色只用于当前动作、重点和错误提示。
- 绿色只表示已连接或成功，黄色表示提醒；不要用颜色代替文字说明。
- 直角或轻微圆角为主，不使用装饰性渐变、光晕和无意义的浮层卡片。
- 标题以层级和留白建立秩序，不使用面板内部的英雄级大字。

## 布局与滚动

- 顶栏和阶段导航是工作区框架；桌面端当前阶段内容独立滚动。
- 输入、方案、结果始终允许切换；没有数据时展示对应空状态，只禁用真正不可执行的操作。
- 长输入、提示词方案、结果历史各自承担滚动，不让多个滚动容器争夺同一手势。
- 方案列表和结果批次列表是主滚动区；标题、筛选和提交操作保持稳定，不随着长列表一起离开视口。
- 固定或粘性区域必须为主要内容保留明确空间，不能遮挡输入、结果或底部操作。
- 桌面端标题允许紧凑化，但必须保留“步骤标识 -> 中文标题”的上下层级；横排只在实际视觉效果更好且不削弱层级时使用。
- 当前产品只支持电脑端；`<= 900px` 只显示边界说明，不渲染或堆叠桌面工作区。

## 标题组件

统一使用 `title-pair` 表达“英文/阶段标识 + 中文标题”。

- 阶段页、蓝图、弹窗共享标识字体、标题字号和间距规则。
- 工作区标题可以比弹窗标题更紧凑，但不能改变语义层级。
- 标题右侧的按钮、状态和数量属于标题操作区，不混入标题文本。
- 新增同类标题先复用 `title-pair`，只有容器宽度或交互职责不同才新增变体。

## 品牌与归属

- 项目归属为 `LINPO LAB`，页面版权为 `© 2026 LINPO LAB. All rights reserved.`。
- 顶栏只显示低权重简写，不增加独立徽章或全局底栏；完整文本放在元数据、电脑端边界页和配置弹窗。

## 控件规则

- 普通按钮使用统一边框、直角、最小高度和禁用透明度；主操作使用橙色填充。
- 图标按钮只用于关闭、折叠、移除等明确动作，必须有 `aria-label` 和 `title`。
- 输入框、文本域和下拉框使用同一边框、背景、聚焦描边和右侧指示器包裹方式。
- 单选框、复选框等选择控件不继承文本输入框的满宽和 40px 高度；控件本体保持稳定尺寸，整行标签承担可点击区域。
- 表单标签、辅助说明和错误说明保持固定的字号层级；辅助文字只解释会改变用户操作的内容。
- 生成数量、状态、批次等选择器必须有真实可用的数据来源，不展示假的未来选项。

## 面板、卡片与弹窗

- 页面主区域使用面板或全宽结构，不在面板内继续套装饰性面板。
- 重复结果使用卡片；卡片内部保持固定的图片、标题、状态和操作顺序。
- 弹窗固定为标题区、内容区、操作区三段结构；危险操作放在次要位置，并明确后果。
- 基础弹窗内容过长时只滚动正文，不让标题和保存/确认操作被滚走；大图和对比弹窗的媒体区可独立滚动。
- API 配置按服务分组，字段顺序统一为接口地址、API Key、模型；移动端不能让清除或危险操作先于保存操作。

## 状态与反馈

- 空状态说明“现在是什么状态、用户下一步做什么、之后会出现什么”。
- 空状态内容作为一个垂直信息组在容器内水平、垂直居中；区域标题和已建立后的结构化内容保持左对齐。
- 加载状态说明当前正在处理的具体动作，并阻止重复提交。
- 错误状态说明失败原因、哪些输入仍然安全、如何重试或修改。
- 弹窗表单校验失败时在字段旁保留错误并聚焦问题字段，不只依赖短暂 Toast。
- 成功反馈使用局部结果或短 toast；不要只改变颜色而没有文字反馈。
- 选中、成功、警告、失败、禁用使用不同语义，不互相借色。

## 历史组件

- 结果历史是高密度浏览界面；批次内统一使用紧凑小卡，不因结果数量或单条批次改变卡片体量。
- 卡片图片区作为缩略预览保持统一高度，完整显示不同画幅比例；查看细节使用大图预览和批次对比。
- 批次标题只负责展开和收起；对比、删除等操作放在独立并列操作区，不嵌套进展开控件。
- 结果卡操作跟随真实状态：成功后才显示“基于此结果细化”，失败时提供恢复动作，加载时避免出现结果型操作。
- 结果历史按用户一次提交形成批次；批次标题同时显示已完成、失败和生成中的真实数量。
- 批次级删除必须明确影响范围，例如删除批次编号及包含的结果数量；单条删除和批次删除都提供短时撤销。
- 历史管理弹窗展示结果数、批次数、带图片记录数和浏览器占用估算；统计只反映当前浏览器本地数据。
- 清理范围使用真实保留数量选项；没有可清理记录时，危险按钮必须禁用并说明原因。
- 清理旧记录或全部清空后，必须在 toast 中说明实际删除数量并提供撤销；撤销应恢复原有顺序和图片清理任务。
- 历史清理不影响当前输入、提示词方案和 API 配置；刷新后仍应保持清理结果。
- `901px` 窄桌面下弹窗内容必须完整可达，操作区保持可见；toast 不得遮挡弹窗内按钮或页面固定操作栏。

## 文案

- 面向创作者使用简体中文；API、URL、Prompt、model 等必须识别的技术词可保留。
- 按钮使用动作词，避免“开始探索”“唤醒引擎”等无法判断结果的表达。
- 首屏要让用户看懂当前模式、必填输入、主操作和结果去向。

## 验收清单

### 验收前置门槛

开始小白视觉验收前，先确认项目存在可引用的设计规范或 UI 风格指南。若不存在：

- 只做单点修复时，可以先记录为“建议制作”，并限定本次改动范围。
- 涉及多个页面、弹窗或重复组件时，应先要求制作项目级规范，再进行大范围视觉统一。
- 没有规范不能直接判定“视觉统一通过”；最多只能报告为“静态审查”或“局部通过”。

规范至少要覆盖页面骨架、标题层级、颜色语义、字体层级、间距、控件、面板/卡片、弹窗、状态反馈和响应式验收口径。

每次页面级视觉调整至少检查：

- 桌面端：标题层级、固定区高度、当前模块滚动、主操作和结果区是否无遮挡。
- 窄桌面：在 `901px` 检查无横向溢出、标题和按钮不裁切、主要操作可见；在 `<= 900px` 检查只出现电脑端边界说明。
- 重复组件：输入、选择器、按钮、图标按钮、标签、卡片、面板、弹窗和 toast 是否仍属于同一套规则。
- 状态闭环：首次空状态、可操作状态、加载、成功、失败、恢复和必要的危险操作。
- 运行证据：控制台无明显错误，主路径可以完成，自动化测试不能替代真实渲染检查。

不因单个间距或颜色问题新增一次性规则；先判断它违反的是布局、层级、组件、状态、文案还是运行边界，再修复共享原因。
